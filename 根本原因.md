这是一个经典的 WSL2 网络隔离问题。让我解释原因并给出解决方案。

## 根本原因

WSL2 运行在一个独立的虚拟机中，它的 `localhost` 和 Windows 的 `localhost` **不是同一个网络命名空间**。Chrome 在 Windows 上监听 `127.0.0.1:9222`，但 WSL2 内部访问 `localhost:9222` 访问的是 WSL2 自己的回环地址，根本到不了 Windows。

## 解决方案

**方案一：把 `localhost` 替换为 Windows 宿主机 IP**

在 WSL2 中获取 Windows 宿主 IP：

bash

```bash
# 获取 Windows 宿主机 IP
cat /etc/resolv.conf | grep nameserver | awk '{print $2}'
```

假设得到的 IP 是 `172.x.x.1`，然后：

bash

~~~bash
claude mcp add --scope user playwright -- npx @playwright/mcp@latest --browser-url http://172.x.x.1:9222
```

同时，Windows 端启动 Chrome 时需要确保监听的不只是 `127.0.0.1`，启动参数加上：
```
chrome.exe --remote-debugging-port=9222 --remote-debugging-address=0.0.0.0
~~~

**方案二：利用 WSL2 的 `localhost` 转发特性（推荐先试）**

较新版本的 WSL2 支持自动将 Windows 的 localhost 端口转发到 WSL2 内。确认你的 `.wslconfig` 中没有禁用这个功能：

检查 `C:\Users\<你的用户名>\.wslconfig`：

ini

```ini
[wsl2]
localhostForwarding=true
```

如果文件不存在或没有这一行，创建/添加后重启 WSL：

powershell

```powershell
wsl --shutdown
```

然后重新进入 WSL，再试原来的命令。

**方案三：在 WSL2 内部直接运行浏览器**

如果上面两种都不稳定，可以考虑在 WSL2 内安装 headless Chrome，这样就不存在跨环境网络问题：

bash

```bash
npx playwright install chromium
claude mcp add --scope user playwright -- npx @playwright/mcp@latest
```

这种方式让 Playwright MCP 自己管理浏览器实例，完全绕开跨网络问题。
